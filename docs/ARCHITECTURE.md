# Архитектура CalcFS PDF Export

## Выбор стека

| Слой | Технология | Обоснование |
|------|--------------|-------------|
| Язык | Python 3.10+ | Быстрая разработка, библиотеки DBF/PDF |
| DBF | `dbfread` | Стабильное чтение без ODBC, memo по возможности |
| PDF | `reportlab` | Программная вёрстка таблиц без браузера |
| Merge | `pypdf` | Чистый Python, предсказуемый merge |
| GUI | `PySide6` | Нативнее Tk на Windows, список с мультивыбором |
| Конфиг профилей | YAML | Человекочитаемое расширение новых отчётов |

## Desktop vs CLI

Реализованы **оба** варианта:

- **GUI** — основной сценарий для оператора на месте проведения.
- **CLI** (`python -m calcfs_pdf_export.cli`) — пакетная выгрузка, скрипты, CI.

## Слой `.rpt` → внутренняя модель

1. **Файл `.rpt`** — диагностируется (`rpt_ole_probe`): OLE, опционально список потоков.
2. **Декларативный профиль** — `templates/report_profiles.yaml` связывает имя `.rpt` с **движком** (`starting_order_v1`).
3. **Движок** — Python-модули, которые знают запрос к DBF и макет PDF.

Так отделяем **наследие Crystal** от исполняемого кода; при появлении спецификации формата можно добавить парсер потоков и заполнять профиль автоматически.

## Поток данных

```
Папка DBF
    → load_calcfs_folder (EVT,CAT,SCP,PAR,PCT,PRF,CLB)
    → discover_cat_scp_pairs (пары из PRF+PAR)
    → пользователь выбирает подмножество
    → для каждой пары: build_starting_order_rows
    → render_starting_order_pdf (временный файл)
    → merge_pdf_files → итоговый PDF
```

## Обработка ошибок

- Ошибка на одной паре **не прерывает** остальные: в `export_pipeline` каждая пара в своём `try/except`.
- Если ни одна пара не успешна, итоговый файл не создаётся, журнал содержит причины.

## Расширяемость

- Новый отчёт: новый модуль сборки строк + функция рендера + запись в `report_profiles.yaml`.
- Общие утилиты: `dbf_utils`, `ids`, `calcfs_store`.
