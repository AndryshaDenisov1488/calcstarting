# Анализ шаблона `StartingOrderWithClubNames.rpt`

## 1. Формат файла

| Признак | Значение |
|--------|----------|
| Размер | 68 608 байт |
| Сигнатура | `D0 CF 11 E0 A1 B1 1A E1` |
| Тип | **Microsoft OLE Compound File** (Structured Storage) |

Такая сигнатура характерна для контейнеров вроде старых документов Office, **Crystal Reports** (классический бинарный `.rpt`) и ряда других продуктов Microsoft / SAP.

**Вывод:** файл **не** является текстовым XML/HTML-шаблоном и **не** является DBF. Его нельзя «прочитать глазами» как CSV или JSON.

## 2. Содержимое на уровне байтов

- Поверхностный поиск ASCII/UTF-16-LE фрагментов с именами полей ISUCalcFS (`CAT_ID`, `PAR_ID`, `PRF_STNUM`, `Crystal` и т.д.) **не дал совпадений**.
- Это указывает на то, что метаданные отчёта лежат внутри **сложных бинарных потоков** OLE (часто со сжатием/упаковкой), а не в виде открытого текста.

**Гипотеза (H1):** движок отчётов CalcFS использует Crystal Reports или совместимый OLE-формат; точная идентификация возможна после разбора каталога потоков (см. модуль `calcfs_pdf_export.rpt_ole_probe` и опциональную зависимость `olefile`).

## 3. Что можно извлечь без проприетарного движка

| Возможность | Статус |
|-------------|--------|
| Список потоков OLE / мини-потоков | Да, с библиотекой `olefile` |
| Полная семантика формул, групп, подотчётов | Нет без спецификации или SDK Crystal |
| Пиксель-в-пиксель идентичный PDF как в CalcFS | Нет гарантии (шрифты, макет, условное форматирование) |

## 4. Семантика отчёта (из имени файла + документации проекта)

Имя **`StartingOrderWithClubNames`** однозначно указывает на бизнес-смысл:

- **Стартовый порядок** → таблица выступлений **`PRF`**, поле стартового номера **`PRF_STNUM`** (см. `docs/OSIS_FULL_INSTRUCTION.md`).
- **Клубы** → связь персоны с клубом **`PCT_CLBID`** и справочник **`CLB.DBF`** (если присутствует в папке соревнования); в OSIS/XML клуб дублируется вложенным блоком (см. `docs/JSON_TYPES_DICTIONARY.md`).

**Assumption (A1):** логика оригинального `.rpt` для этого имени эквивалентна отчёту «по выбранной паре (категория × сегмент): список участников в порядке стартовых номеров с названием клуба».

## 5. Связанные таблицы DBF (ожидаемый минимум)

| Таблица | Роль в стартовом списке |
|---------|-------------------------|
| `EVT` | Название соревнования (шапка) |
| `CAT` | Категории; ключ `CAT_ID` |
| `SCP` | Сегменты; ключ `SCP_ID`; название сегмента |
| `PAR` | Участник в категории; `PAR_ID`, `CAT_ID`, `PCT_ID` |
| `PCT` | Персона/пара; ФИО, `PCT_CLBID`, страна |
| `PRF` | Выступление в сегменте; `PAR_ID`, `SCP_ID`, `PRF_STNUM` |
| `CLB` | Клуб по `PCT_CLBID` (если файл есть) |

## 6. Индексы CDX

Чтение через `dbfread` **не использует** CDX: порядок записей — как в DBF-файле. Для стартового порядка сортировка задаётся явно по **`PRF_STNUM`** (и при равенстве — по `PAR_ID` / `PRF_ID` для стабильности).

**Риск:** если в CalcFS порядок «официального» отчёта завязан на сложный индекс, поведение может отличаться; для стартового листа обычно достаточно `PRF_STNUM`.

## 7. Параметры отчёта в терминах данных

Вместо скрытых параметров Crystal в нашем MVP:

- **Категория** — `CAT_ID`
- **Сегмент** — `SCP_ID`
- Набор пар `(CAT_ID, SCP_ID)` выбирается в GUI; по каждой паре строится отдельный PDF, затем merge.

## 8. Можно ли воспроизвести без CalcFS

| Часть | Без CalcFS |
|-------|------------|
| Данные | Да, при наличии копии папки DBF |
| Точный бинарный `.rpt` | Нет (нужен Crystal/встроенный движок CalcFS) |
| Смысловой эквивалент стартового списка | Да (см. реализацию `starting_order_report`) |

## 9. Риски и ограничения

1. **Проприетарный `.rpt`** — нет открытой спецификации на уровне полей/формул.
2. **Кодировка DBF** — часто CP1251; возможны CP866 и искажения имён (fallback кодировок в коде).
3. **Memo (FPT)** — при отсутствии файла memo поля могут быть пустыми (`ignore_missing_memofile`).
4. **Имена полей** — в разных версиях CalcFS возможны отличия; в коде используется **гибкое сопоставление** нескольких кандидатов имён.
5. **Пары/танцы** — отображение как одна строка по `PCT_CNAME` или склейка `PCT_GNAME` + `PCT_FNAME` (**assumption A2**).

## 10. Промежуточный слой шаблона

Файл `templates/report_profiles.yaml` связывает:

- человекочитаемое имя отчёта (и при желании путь к `.rpt` как **метаданные**),
- **движок** Python (`starting_order_v1`),
- опции (колонки, сортировка).

Так мы отделяем «наследие Crystal» от исполняемой логики, сохраняя трассируемость к исходному `.rpt`.
